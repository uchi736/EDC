01 — 目的

目的とスコープ

社内技術文書（不適合報告書・規程・規格・技術文書 等）から、後段の RAG / QA / 分類で使える 知識グラフ（トリプル）を、外部サービスに一切送信せず構築する。

  • 完全オンプレ — LLM・埋め込み・OCR をすべて社内 DGX Spark 上で稼働（外部送信なし）。
  • 妥当かつ最小限のエッジ — 文書タイプ別の標準スキーマ ＋ 半自動拡張 ＋ 類似度マージで、スキーマの際限ない肥大化を防ぐ。
  • 半自動化（HITL） — 精度改善のため、スキーマを人がレビュー／キュレーションできる。
  • 外部から呼べる — 他 RAG アプリ / Dify から HTTP(API) で呼べる。

02 — 構成

全体アーキテクチャ

アプリ層は LLM・埋め込み・OCR をすべて HTTP で DGX Spark に委譲する。よってアプリ層は軽量 （torch/transformers 不要）。3つのアクセス面が同じコア（edc/）を共有する。

03 — フロー

処理パイプライン

入力から知識グラフまで。前段（タイプ分類）・EDC 本体（E→D→C）・ 連結層（名寄せ）の3部構成。EDC 本体は上流実装を無改変で利用。

ステージ実装入力 → 出力
OCRpdf_processorPDF → ページ毎テキスト（PaddleX）
Extract (OIE)edc/extract.pyテキスト → 生トリプル（3/5タプル）
Defineedc/schema_definition.py生関係 → 定義
Canonicalizeedc/schema_canonicalization.py生関係 → 確定スキーマへ写像
Type Canon.edc/type_canonicalization.py生型 → 確定型へ写像

04 — スキーマ

スキーマ設計（3層）

「定型を固定し、それ以外を半自動で足す」設計の核。不変骨格 ＋ 文書タイプ別ドメイン層 ＋ 連結層。

レジストリ（schemas/registry.json）が「文書タイプ→説明＋ドメインCSV」を保持し、 ルーターが 不変骨格 ∪ ドメイン層 を合成して EDC に渡す。新しいクエリ型が必要な新文書タイプが来たときだけ昇格・追加する。

05 — 前段

前段ルーター（文書タイプ分類）

LLM の役割は「どのタイプか」の分類のみ。スキーマ設計はさせず、事前固定した標準スキーマへルーティングする。

06 — 正規化

類似度マージゲート

スキーマの際限ない肥大化を防ぐ中核ロジック（関係・型で対称）。閾値 τ = CANON_MERGE_THRESHOLD（既定 0.9）。

force-fit 注意 — ドメインに合わない汎用すぎる初期スキーマを当てると、本来別概念がコア関係に押し込まれ精度が落ちる。→ 文書タイプ別ドメイン層で緩和する。

07 — HITL

Human-in-the-loop（スキーマ・キュレーション）

EDC 本体を Phase A / Phase B に分割し、間に人手レビューを挟む（app.py）。関係スキーマと型スキーマを対称にキュレーションする。

08 — API

アクセス面と API

CLI / Streamlit / HTTP API の3面が同じコアを共有。外部連携は FastAPI（OpenAPI 自動公開）。 Dify は /openapi.json をカスタムツールとして取り込み /extract を呼ぶ。

09 — デプロイ

デプロイ / コンテナ

イメージはスリム（requirements-api.txt: openai/requests/numpy/tqdm/dotenv/fastapi/uvicorn のみ）。 LLM/埋め込み/OCR は DGX 側のまま。.env はイメージに焼き込まず実行時注入。

13 — 今後

既知の制約・今後

  • 条件付きエッジ — 「金額≥1000万→役員承認」のような閾値付き条件は素の 3/5 タプルでは構造化しきれない。条件ノード化（Rule -APPLIES_WHEN-> Condition{値,演算子}）かエッジ属性が必要。
  • チャンク分割 — フォーム文書は「ラベル↵値」が分断され退化トリプル（自己ループ等）が出る。ラベル:値の束ね／見出し・再帰分割で改善余地。
  • 範囲外（後段レイヤ） — 同名異義の曖昧性解消（WSD）・専門用語辞書アンカー・グラフDB格納/スケール。
  • refinement（反復改善） — CLI/EDC 本体に存在するが、API(one-shot)では未使用。
