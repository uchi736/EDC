Extract · Define · Canonicalize

社内文書から、知識グラフを
自動でつくる仕組み

不適合報告書・規程・技術文書などの非構造テキストを、LLMで「主語 → 関係 → 目的語」のトリプルに変換します。データは一切外部に出さず、すべて社内サーバー上で完結します。

📄 文書 → Extract 抽出 → Define 定義 → Canonicalize 正規化 → 🕸 知識グラフ
01

中核：文書の1文が知識グラフになるまで

EDC は3つのステージを順に通します。ここでは実際の不適合報告書の1文を追いかけ、各ステージで何が起きるかを見ます。

入力の1文： 溶接部の予熱不足が低温割れの原因となった。 — この文をパイプラインに流します。

Stage 1 · Extract

抽出（OIE）

文からエンティティと関係を、スキーマにとらわれず“素のまま”取り出す。

出力：素のトリプル

raw triple
予熱不足 ―原因となった→ 低温割れ

Stage 2 · Define

定義

抽出された関係が「何を意味するか」を LLM に一文で定義させる。

出力：関係の定義文

definition
原因となった：主語が目的語を引き起こす／発生の原因となる関係。

Stage 3 · Canonicalize

正規化

定義をもとに埋め込み検索＋LLM検証で、標準スキーマの関係へ寄せる。

出力：確定トリプル

canonical triple
予熱不足:Cause ―CAUSES→ 低温割れ:Defect
02

スキーマは3層で「増えすぎ」を防ぐ

正規化の寄せ先となる関係の一覧（スキーマ）は3層構造。定型は固定し、それ以外だけを半自動で足すことで、関係が際限なく増えるのを抑えます。

1不変骨格

全文書で共通・手で固定

文書タイプに依存しない土台。必ず含める関係の閉集合。

DEFINESMENTIONS REFERS_TOIS_APART_OF

2ドメイン層

文書タイプ別・昇格制

NCR・規程・論文など、タイプ固有の関係。新タイプ登場時だけ追加。

CAUSESMITIGATES APPLIES_TOREQUIRES_APPROVAL_FROMDEPENDS_ON

3連結層

名寄せ・エッジにしない

同一・別名はエッジではなくノード統合で処理（1実体=1ノード）。

SAME_AS → マージALIAS_OF → 別名属性

前段のルーターが文書タイプを判定し、「不変骨格 ∪ そのタイプのドメイン層」を合成して正規化の寄せ先に渡します。

03

正規化の判断ロジック：類似度マージゲート

抽出された関係を1つずつ、この分岐にかけます。「初期スキーマに寄せる／近い既存に統合する／本当に新しいものだけ追加する」を自動で振り分け、スキーマの肥大化を防ぐ中核です。

生の関係／型が入力される

例：原因となった

確定スキーマに既にある？

Yes → そのまま採用No → 次へ

埋め込み top-k 検索 ＋ LLM が候補から合致を選ぶ？

意味の近い既存の関係を探し、LLMが「これと同じか」を検証

Yes → 既存へマッピングNo → 次へ

最近傍との類似度 cos ≥ τ（既定 0.9）？

閾値以上なら「言い換えレベルに近い」とみなす

Yes → 最近傍へ統合（近傍同義を吸収）No → 次へ

本当に新しい概念だけを判定

enrich オン → 新規スキーマとして追加オフ → 除外 / Other

実測では、フリー発見で 95 種 に膨らんだ関係が、ドメイン初期スキーマ＋このゲートで 24 種 に収束しました。

04

3つの入り口が、同じコアを共有

バッチ処理・人手レビュー・外部連携の3つの使い方があり、いずれも同じ EDC コア（E→D→C）を呼びます。

run.py

CLI バッチ

フォルダ内の .txt/.pdf/.md をまとめて処理し、triplets.json を書き出す。

app.py

Streamlit（HITL）

抽出→人がスキーマをレビュー・修正→正規化。精度を人手で高める2フェーズ運用。

api.py

FastAPI（:8080）

Dify や他 RAG アプリから POST /extract で呼べる HTTP API。

05

完全オンプレ：データは社内から出ない

アプリ層は軽量で、LLM・埋め込み・OCR という重い処理は社内 DGX Spark に HTTP で委譲します。外部サービスへの送信はゼロです。

アプリ層（軽量コンテナ）

api.py / run.py / app.py

EDC コア＋文書タイプ・ルーター。torch 不要のスリム構成。

pdf_processor.py

PDF はOCRへ、txt/md はチャンク分割。

→ LAN 内
HTTP のみ

DGX Spark（社内・完全ローカル）

vLLM · Chat LLM

:8000

抽出・定義・正規化検証を担う LLM。

vLLM · Embedding

:8003

関係・型の類似度検索用の埋め込み。

PaddleX · OCR

:8005

PDF をページ単位でテキスト化。

ポイント：アプリ層は「頭脳」を持たず、すべての推論を社内サーバーへ委譲する薄い殻。だからコンテナは軽く、機密文書も社外に一切出ません。

06

出てくるもの：2種類のトリプル

最終成果は知識グラフのトリプル。既定は3タプル、typed モードでは型付きの5タプルになります。

3タプル ・ untyped（既定）

主語 → 関係 → 目的語

予熱不足 ― CAUSES → 低温割れ

5タプル ・ typed モード

エンティティに型が付き、型も正規化される

予熱不足Cause ― CAUSES → 低温割れDefect

出力は triplets.json（ファイル別トリプル）と discovered_schema.csv（発見した関係と件数）。API では JSON で返ります。
