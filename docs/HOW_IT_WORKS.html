Extract · Define · Canonicalize — 実装ロジック解説

社内文書から知識グラフをつくる、その中身

非構造テキストを「主語 → 関係 → 目的語」のトリプルに変換するパイプラインを、実際のコードのロジック（正規化の分岐・類似度ゲート・typed処理・反復改善・前段ルーター・堅牢化）まで踏み込んで図解します。データは一切外部に出さず、社内サーバーで完結します。

📄 文書 / PDF→ 🧭 タイプ分類→ Extract→ Define→ Canonicalize→ 🕸 知識グラフ
01

中核：1文が知識グラフになるまで（E→D→C）

EDC 本体（edc/edc_framework.py）は3ステージを順に通します。各ステージは専用のプロンプトテンプレート＋Few-shot例でLLMを駆動。実際の不適合報告書の1文を追いかけます。

入力の1文： 溶接部の予熱不足が低温割れの原因となった。

Stage 1 · Extract (OIE)

抽出

スキーマにとらわれず“素の”トリプルを取り出す。edc/extract.py

  • プロンプト＋Few-shot例でLLMに抽出させる
  • parse_raw_triplets が括弧対応で頑健にパース。3or5要素・空/_は除外
  • typed時は専用プロンプトで5タプルを直接出力
raw triple
予熱不足 ―原因となった→ 低温割れ

Stage 2 · Define

定義

出現した各関係の意味を一文で定義。edc/schema_definition.py

  • 1文中のユニークな関係を集約し、まとめて定義生成
  • parse_relation_definition が 関係: 定義 行を辞書化
  • この定義文が次段の埋め込み検索キーになる
definition
原因となった：主語が目的語を引き起こす関係。

Stage 3 · Canonicalize

正規化

定義をもとに標準スキーマの関係へ寄せる。edc/schema_canonicalization.py

  • 定義文を埋め込み、top-5の近い既存関係を検索
  • LLMが選択肢から合致を選ぶ（or 該当なし）
  • 該当なしでも類似度ゲートで統合／追加を判定（§04）
canonical triple
予熱不足:Cause ―CAUSES→ 低温割れ:Defect
temperature=0（決定的） provider: vllm / azure / openai / local stage毎に個別のprompt＋few-shot
02

正規化の判断ロジック（詳細）：類似度マージゲート

これが「初期スキーマに寄せる／近い既存に統合する／本当に新しいものだけ追加する」を自動で振り分ける中核です（SchemaCanonicalizer.canonicalize）。分岐は上から順に評価されます。

生の関係が入力される　例：原因となった

① すでに確定スキーマに存在する関係名？

open_relation in schema_dict の単純一致チェック

Yes → そのまま採用（LLM呼ばない）No → ②へ

② スキーマが空 or 定義が無い？

スキーマフリー（初期スキーマ未指定）や、Defineが定義を出せなかった関係は検証をスキップし ⑤ へ

該当 → 候補なしで ⑤ へそれ以外 → ③へ

③ 埋め込み top-5 検索 ＋ LLM検証

retrieve_similar_relations：関係名でなく定義文を正規化コサイン類似度で比較し上位5件を取得。LLMには A. B. C. … ＋ None of the above の選択肢を提示し、1文字で回答させる（llm_verify）

合致を選択 → 既存関係へマッピングNone of the above → ④へ

④ 最近傍との類似度 cos ≥ τ ？

LLMが「該当なし」でも、最近傍(scores[0])が閾値以上なら“言い換えレベル”とみなし統合。近傍同義を吸収してスキーマ肥大を抑える

Yes → 最近傍へ統合No → ⑤へ

⑤ enrich（漏れ追加）モード？

本当に新しい概念だけをここで追加。追加時は定義をその場で埋め込み、以降のマッチ対象に動的登録するため、同一文書内の後続トリプルが新関係にマッチできる

enrich=ON → 新規スキーマとして追加OFF → 除外（トリプルを捨てる）
τ = CANON_MERGE_THRESHOLD（既定 0.9） top_k = 5 埋め込み対象 = 関係の定義文 類似度 = 正規化コサイン

効果：フリー発見で 95種 に膨らんだ関係が、ドメイン初期スキーマ＋このゲートで 24種 に収束（実測）。「初期を与えて漏れは追加、ただし際限なく増やさない」を実現します。

03

Typed モードの内部処理（5タプル化と型正規化）

エンティティに型を付ける typed モードは、target_types_path 指定か enrich_types で有効化。ここが最もトリッキーで、5タプルを一旦3タプルへ縮約→SD/SC→型を再付与→型を正規化という順で処理します（SD/SCは3タプル前提のため）。

1. Typed OIE が5タプルを出力

[予熱不足, Cause, 原因となった, 低温割れ, Defect]

2. 3タプルへ縮約（型はサイドカー退避）

[t[0], t[2], t[3]] で [予熱不足, 原因となった, 低温割れ]。元の5タプルは typed_oie_list に保持

3. Define → Schema Canonicalization（関係の正規化）

§02 のゲートで 原因となった → CAUSES

4. 型を位置合わせで再付与

SC後の3タプルに、退避した subj_type / obj_type を index で戻し5タプルへ復元

5. Type Canonicalization（型の正規化）

TypeCanonicalizer は §02 と対称なゲートで Cause / Defect を確定型へ寄せる。(entity, raw_type) をキャッシュして重複LLM呼び出しを削減。確定できない型は "Other"

型ゲートも τ・top_k=5 で対称 キャッシュ: (entity, raw_type) → type 未確定 → "Other" untyped時は3タプルを素通し
04

反復改善（Refinement）：ヒント付き再抽出

--refinement_iterations N を付けると、1回目の結果を手がかりに N 回まで再抽出します。表面的に取り逃した関係を拾う仕組み。CLI/EDC本体のみで、API(one-shot)では未使用。

ヒントの組み立て

construct_refinement_hint

前回のトリプルからエンティティ／関係を収集

EntityExtractor で候補エンティティ抽出

前回分と merge_entities で統合 → entity hint

SchemaRetriever で関連関係を検索

入力文を埋め込み、スキーマから top-10 の関連関係を取得。例文付きで relation hint 化

refined OIE を entity/relation hint 付きで再実行

専用テンプレート oie_r_template

反復ループ全体

extract_kg の for iteration in range(N+1)

  • 各 iteration で OIE → Define → Canonicalize を実行
  • 結果を次イテレーションの hint として渡す
  • 反復ごとに iter{i}/ ディレクトリへ中間結果を出力
  • 不要になったモデルは free_model でGPU解放（local時）
別embedder：Retriever は SC とは別の sr_embedder を使い、専用の指示プロンプトで文→関係を検索します。
05

前段ルーター：文書タイプ分類 → 標準スキーマ合成

--doctype auto 時、EDC本体を呼ぶ前に文書を分類し、該当タイプの標準スキーマを自動適用します（doctype_router.py）。LLMの役割は分類のみで、スキーマ設計はさせません（事前固定）。

1. 分類（classify_document）

registry のタイプ一覧＋サンプルをLLMに提示

# temperature=0, JSONのみ出力を要求
{ "type": "NCR_不適合報告書",
  "confidence": 1.0,
  "reason": "是正処置・原因記載あり" }

_parse_json_obj がコードフェンス除去・最初の {…} 抽出で頑健にパース。失敗や該当なしは UNKNOWN。

2. スキーマ合成（compose_target）

不変骨格 ∪ ドメイン層 を一時CSVへ

registry から骨格CSVとドメインCSVを読む

辞書マージ（同名はドメイン優先で上書き）

relation用とtype用の両方を合成

一時CSVを書き出し、そのパスをEDC設定へ

target_schema_path / target_types_path ＋ enrich=ON

UNKNOWN のとき：compose_target は (None, None) を返し、従来設定（--target_schema_path 指定 or フリー発見＋HITL）にフォールバックします。

06

HITL：抽出と正規化を分け、間に人手レビュー

Streamlit UI（app.py）は EDC を2フェーズに分割し、発見スキーマを人がキュレーションできます。

Phase A — extract_and_define()

OIE + Define のみ（ファイル出力なし）

  • 発見スキーマを構築：定義は最長のものを採用、count はOIE出現回数
  • typed時は発見型リストも同様に構築（初期型定義をシード）
  • st.data_editor で人にスキーマ提示 → 統合/リネーム/定義修正/採用・削除

Phase B — canonicalize_with_schema()

人が確定したスキーマで再正規化

  • self.schema = curated_schema を設定して SC 実行
  • SchemaCanonicalizer は毎回 self.schema を再embedding → 確定スキーマへマッピング
  • typed時は型の再付与＋Type正規化も対称に実行
  • enrich_types で「初期型＋漏れ追加」も選べる
07

スキーマ3層で「増えすぎ」を防ぐ

正規化の寄せ先スキーマは3層構造。定型を固定し、それ以外だけを半自動で足す設計。レジストリ(schemas/registry.json)が「タイプ→ドメインCSV」を保持します。

1不変骨格

全文書で共通・手で固定

タイプに依存しない土台。必ず含む関係の閉集合。

DEFINESMENTIONSREFERS_TOIS_APART_OF

2ドメイン層

文書タイプ別・昇格制

NCR・規程・論文などタイプ固有。新タイプ時だけ追加。

CAUSESMITIGATESAPPLIES_TOREQUIRES_APPROVAL_FROMDEPENDS_ON

3連結層

名寄せ・エッジにしない

同一・別名はノード統合で処理（1実体=1ノード）。

SAME_AS → マージALIAS_OF → 別名属性

プロバイダ抽象と堅牢化

edc/utils/llm_utils.py がLLM/埋め込みの差異を吸収。同じコードで vLLM・Azure・OpenAI・ローカルHFを切り替えます。

仕組み内容

リトライ上限API失敗時は5秒待って再試行。VLLM_MAX_RETRIES（既定5）で打ち切り → エンドポイント停止時も無限ハングしない
タイムアウトクライアントに VLLM_TIMEOUT（既定300秒）を設定
決定性全LLM呼び出しが temperature=0
頑健パース括弧対応でトリプルを抽出、型不整合や空要素を除外。JSON抽出もフェンス除去付き
遅延ロードtransformers/SentenceTransformer はローカルモデル時のみ import（APIコンテナは torch 不要で軽量）
